# Routes And Pages

## Current State After Phase 7

Registered route metadata now flows from `src/lib/wiki/static-pages.ts` or class/species domain records into `src/lib/wiki/registry.ts`. The root layout owns document metadata and `PageHeader` for static/information/rules/search/preference/legal pages. `src/routes/classes/+layout.svelte` and `src/routes/species/+layout.svelte` are the nearest metadata owners for class/species route chains and render availability badges by canonical href.

Current route pages should contain body content only. Structural tests reject child-route `PageHeader` imports, route-local `<svelte:head>` blocks, route-local h1s under header-owning layouts, and route-local class/species availability declarations.

## Historical Notes From Original Audit

- `/classes` routes were already close to layout-owned PageHeader.
- `/species` routes, top-level legal/information routes, and rules child lists had route-local metadata duplication in the original audit snapshot.
- Those items are no longer current editing guidance. Phase 4 moved static route metadata to the registry flow, and Phase 7 structural tests keep route pages body-only for registered pages.

## Phase 4 Route Metadata Result

Phase 4 moved static, information, legal, rules, monsters, locations, search, preferences, and home route metadata behind the registry flow. The root layout now derives the active registry entry by href and owns document metadata plus `PageHeader` for non-class/species pages. The class and species layouts remain the nearest active metadata owners for those domains.

Route body files no longer own structural page metadata. They should not import `PageHeader`, render `AvailabilityBadges`, declare route-local `svelte:head`, or define header-only title/description/eyebrow/tag objects. Body sections, rules prose, tables, notices, filters, and route-specific interactive state remain route-owned.

Rules child cards now derive from `getNavigationChildren('/rules')`. Home quick links remain curated, but the curation is a list of registry page IDs; title, href, and description come from `getWikiPage()`. Footer links derive from registry page IDs through `getFooterPages()`.

Route existence is now covered by structural tests: registered physical pages must resolve to `src/routes/**/+page.svelte`, and physical Wiki routes must have registry metadata. Href changes still require a matching SvelteKit route-folder rename.
